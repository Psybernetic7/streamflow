# Repo Orientation: StreamFlow

> Generated by /orient. Re-run to update.

## One-line purpose
A browser-based torrent streaming app that combines a React frontend, a Node.js backend (WebTorrent + FFmpeg HLS transcoding), and an optional Electron desktop wrapper — letting users search for and play movies/TV shows directly from torrent swarms.

## Primary language(s)
JavaScript (ESM throughout — no TypeScript)

## Pipeline / workflow stages
1. **Discovery** — user searches via TMDB (`/api/search/tmdb`) to find movie/show metadata and poster art
2. **Torrent search** — frontend calls `/api/search/torrents`, backend fans out to Knaben, PirateBay, and SolidTorrents
3. **Metadata fetch** — clicking a torrent triggers `/api/info` (SSE); backend tries to fetch a `.torrent` file from public caches, falls back to magnet + DHT
4. **WebTorrent add** — torrent is added to the shared client with sequential download strategy; SSE stream reports progress until `ready` fires
5. **Streaming** — two modes: direct HTTP range (`/api/stream`) for natively supported formats, or HLS transcode (`/api/hls/*`) via FFmpeg for everything else
6. **HLS transcode** — FFmpeg reads from a WebTorrent file pipe, writes `.ts` segments + `.m3u8` playlists to a tmpdir session; sessions expire after 10 min of inactivity
7. **Playback** — `StreamPage.jsx` uses `hls.js` to play the HLS stream, polls `/api/stats` for live peer/speed stats, and handles subtitle tracks via OpenSubtitles or embedded extraction

## Key files
- `streamer/server.js` — the entire backend in one file: Express routes, WebTorrent client lifecycle, HLS session management, TMDB proxy, subtitle search | read this to understand how any backend feature works
- `electron/main.js` — Electron wrapper that spawns the backend as an in-process module and opens it in a BrowserWindow; resolves platform-specific FFmpeg binary paths | read this to understand the desktop packaging model
- `frontend/src/App.jsx` — React router root; maps URL paths to page components | quickest way to see all top-level pages
- `frontend/src/pages/StreamPage.jsx` — the most complex frontend file: HLS player, audio/subtitle track switching, download-progress overlay, torrent stats polling | central to understanding the playback UX
- `frontend/src/api/torrents.js` — all frontend → backend API calls; `getTorrentInfo` uses `EventSource` (SSE), everything else is plain `fetch` | the contract between frontend and backend
- `ffmpeg/linux/` — bundled FFmpeg/FFprobe binaries; Electron reads paths from here via `getBinaryPaths()` in `main.js`

## Core concepts

**SSE info flow**: `/api/info` is a Server-Sent Events endpoint, not a simple JSON call. It emits `status` events while work is in progress (fetching metadata, waiting for peers) and a single `ready` event with the file list when done. `getTorrentInfo()` in `torrents.js` wraps this with a `Promise` + `EventSource`.

**HLS session**: Stored in the `hlsSessions` Map in `server.js`, keyed by `infoHash:fileIndex`. When a client requests `master.m3u8`, `getOrStartHlsSession()` starts FFmpeg, which pipes from the WebTorrent file stream into a tmpdir. Segments are served as static files. Sessions auto-destroy after 10 min idle (`HLS_TTL`).

**Hardware encoder detection**: `detectHwEncoder()` runs once on first HLS request and caches the result. It tries nvenc → vaapi → qsv by doing a real one-frame test encode (not just checking `ffmpeg -encoders`) and falls back to `libx264` if all fail.

**fetchTorrentBuffer**: Before joining a swarm via magnet+DHT (slow), the backend tries to grab a pre-made `.torrent` file from public caches (itorrents.org, thetorrent.org, torrage.info). A valid `.torrent` is bencoded and always starts with byte `0x64` (`d`) — this is the validation check at line ~657.

**TMDB proxy + cache**: All TMDB API calls go through `tmdbFetch()` which uses a TTL-based in-memory cache and a `coalesce()` helper to deduplicate concurrent identical requests. The TMDB key is hardcoded as a fallback but can be overridden with `TMDB_API_KEY`.

## Common gotchas

1. **Invalid torrent identifier after successful fetch** — `torrage.info` (and sometimes other sources) returns an HTML error page with HTTP 200. The code filters these out by checking `buf[0] === 0x64`, but if you add a new torrent cache source, make sure it also passes this check before being handed to WebTorrent.

2. **HLS "took too long to produce first segment"** — the polling loop in `getOrStartHlsSession()` gives FFmpeg 30 seconds (60 × 500ms) to write the first `.m3u8`. The most common causes: (a) a hardware encoder was detected but its GPU driver is absent, causing FFmpeg to exit silently; (b) the torrent file stream isn't buffered yet. Check the `[hls]` log lines to see which encoder was selected.

3. **Single shared WebTorrent client, max 5 torrents** — `getClient()` returns a singleton; `evictOldest()` removes the oldest torrent when the cap is hit. If you see a torrent disappear mid-stream, it was evicted. The TTL is 30 min (`TORRENT_TTL`) — changing this affects memory usage significantly.
